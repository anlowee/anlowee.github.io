### [POJ3253]
**[👉CLICK HERE](http://poj.org/problem?id=3253)**
#### Description
Given an integer $N(1 \leq N \leq 20000)$, and there are $N$ sticks, whose length $L_i(1 \leq L_i \leq 50000)$ is also an integer. Now you need to compund them into one single long stick, each time you can compound two sticks that will costs you the sum of two sticks' length. You need to **find a way that costs least**.

#### Analysis
Obviously, the order you operate will make a difference. Since it only can operate two sticks each time, we can easily get **binary tree** is a good data structure. Further more, considering its special feature, we can get that **Huffman Tree** is completely a feasible way of this puzzle. 

According to the feature of the Huffman tree, the weighted path of the entire tree is the **smallest**. The method of calculating the weighted path at a vertix is the **product** of the path length from the root node to the vertix and the weight of the vertix. It is not difficult to find that the entire merging process is regarded as **the construction process of a binary tree**. The merging of two sticks is the **parent** node of the two sticks. When a stick is merged for the first time, it obviously that **its value will contribute to the answer again when its parent node participates in the merge**, until the final stick, the root node. When the answer is the smallest, it coincides with the feature of the Huffman Tree. Therefore, this problem is to simulate **the construction process of Huffman Tree**.

To do this, we first put all the sticks into an ascending priority queue. Take the first **two** sticks from the priority queue each time. **When there is only one stick left in the queue, it means that all the sticks have been merged**. At this time, the remaining stick in the queue is the last long stick, and the algorithm ends. After taking out two sticks, merge the two sticks, and then put the merged sticks into the priority queue. Obviously, **the number of sticks in the priority queue is reduced by one after each operation**, so the algorithm must be terminated. Obviously, the optimal answer can be obtained after constructing in this way, because the wooden sticks with a **small length** are **merged first**, and the wooden sticks with a **long length** are **merged later**. In this way, the **depth** of the wooden sticks with a longer length is smaller than that of the wooden sticks with a small length. The coefficients for weighted paths are smaller, and the final answer is certainly smaller. In fact, a **greedy** idea is used here, which is also the basic construction method of Huffman Tree.

#### Algorithm
+ Create a new **ascending priority queue** and **put all the initial sticks in it**.
+ **Determines** whether the number of elements in the priority queue $size$ is greater than $1$. If so, the first two elements of the queue are taken out, otherwise the algorithm ends.
+ The two elements taken out are combined, and the added weights are added to the answer and put in the priority queue.
+ **Repeat** the second step.

#### Code(Java)
~~~Java
import java.util.Scanner;
import java.util.PriorityQueue;
import java.util.Queue;

public class Main {
    public static void main(String[] args) {
        
        Scanner sc = new Scanner(System.in);

        Queue<Integer> pq = new PriorityQueue<Integer>();

        // read data and put them into priority queue
        int n = sc.nextInt();
        for (int i = 0; i < n; i++) {
            int weight = sc.nextInt();
            pq.offer(weight);
        }

        // take two sticks from queue each time
        long ans = 0;
        while (pq.size() > 1) {
            int weight1 = pq.poll();
            int weight2 = pq.poll();

            // add sum into queue
            pq.offer(weight1 + weight2);
            ans += weight1 + weight2;
        }

        System.out.println(ans);
    }
}
~~~

### [POJ1182]
**[👉CLICK HERE](http://poj.org/problem?id=1182)**
#### Description
Given $N(1 \leq N \leq 50000)$ kinds of animals and $K(1 \leq K \leq 100000)$ instructions, each instruction contains three number, the first number is either $1$ or $2$, and the two following are $x$ and $y$. If the first one is $1$, it means $x$ and $y$ are of same kind, else it means $x$ is the predator of $y$. There are totoally three kinds, $A, B, C$, in which $A$ prey on $B$, $B$ prey on $C$, $C$ prey on $A$. Your mission is work out how many wrong instrucions are there, which are **contradictory with previous instructions** or **$x > n, y > n$** or **$x$ eat $x$**.

#### Analysis
First, because there are three sets, and there are a large number of operations that involve associating two vertixs and querying the relationship between two vertixs, it is easy to think of using **union-find set** to maintaining data. The union-find set supports two operations of merging two subsets and querying whether two vertixs are in the same set. And because **the path can be compressed** when looking for the root node of the collection, the algorithm is very efficient.

The standard node of union-find set only contains information about its parent node. According to the intent, there are three kinds of relationships between any two animals: they are of same kind, A preys on B, and B preys on A. Therefore, it is **not enough** to just maintain the parent node's information. It also needs to **maintain the relationship with the parent node(actually with root node)**. It is not difficult to find that the predatory relationship of the three sets A, B, and C constitutes a **cycle**, so we can **use $0, 1, 2$ to represent these three relationships**, and then we will analyze why do this.

To improve efficiency, **path compression** is essential. But when the path is compressed, we **cannot just change the parent node to the root node**, because we also need to maintain the relationship between the current vertix and the parent node. Therefore, the parent node changes, and **this relationship needs to be changed simultaneously**. Assuming the current node is $cur$, the parent node is $par$, and the grandpa node is $gra$, **the relationship between $cur$ and $gra$ is $rel(cur) + rel(par) \models 3$**, where $rel(x)$ represents the relationship between $x$ and its parent node (the range is $0,1,2$), and $\models$ is the modulo operation. **For example**, $cur$ predator $par$, $par$ predator $gra$, the calculation result is $(1 + 1) \models 3$, the answer is $2$, that is, $cur$ is predated by $gra$, which is obviously In line with the question. It is not difficult to verify that such a relational operation is correct. My understanding is that the operation of this relationship becomes relative after modulo. **The relationship of $a$ to $b$ plus $b$ to $c$ is naturally equal to $a$ to $c$. This can be thought of as the absolute speed in physics is equal to the relative speed plus the involved speed**.

In this way, the problem of merging two vertixs is solved. For queries, it's even easier. Because when the path is compressed as described above, the node does not actually save the relationship with the parent node, **but the relationship with the root node** of the collection. So for the two vertixs to be queried, **first see if their root nodes are the same**. If the root nodes are different, it means that they have nothing to do with before, so there will be no problem whether you perform the first operation or the second operation. If the root nodes are the same, **compare their respective relationships with the root node**. For example, there are two vertixs $a$ and $b$. Their relationship with the root node of the set is $rel(a)$ and $rel(b)$. The relationship between $a$ and $b$ is the relationship between $a$ and the root node plus the relationship between the root node and $b$, which is $rel(a) - rel (b) \models 3$. The reason why it is $-rel(b)$ is because the relationship between $b$ and the root node and the relationship between the root node and $b$ are opposite. **If this relationship is different from the current instruction number $cmd - 1$, then it is contradictory**.(When the number of instructions $cmd$ is $1$, it corresponds to two animal are of same kind, so minus one can correspond to it, and the same is true for $2$).

#### Algorithm
+ Initialize the union-find set.
+ Remove the instructions whose parameters do not meet the conditions first.
+ Query the root node of two animals, and do path compression at the same time to find the relationship between the root node and the animal.
+ If the root nodes of the two animals are different, then merge directly and set the relationship. If the root nodes of the two animals are the same, do the next step.
+ Calculate the relationship between two animals. If it is not equal to the number of instructions minus one, the instruction must conflict with the previous instructions, and the answer plus one.

#### Code(Java)
~~~Java
import java.io.BufferedReader;
import java.io.InputStreamReader;
import java.io.IOException;
import java.io.StreamTokenizer;
import java.io.OutputStreamWriter;
import java.io.BufferedWriter;
import java.io.PrintWriter;

public class Main {

    private static class UnionFindSetNode {

        private int parent;
        private int relationship; // relationship with ancestor, 0-of same kind, 1-ancestor's predator,
                                  // 2-ancestor's prey

        public UnionFindSetNode(int index) {
            // initialize node
            parent = index;
            relationship = 0;
        }
    }

    private static final int SIZE = 50005;
    private static UnionFindSetNode[] unionFindSetNodes = new UnionFindSetNode[SIZE];
    // use buffered stream to read and write, or it will TE, casuse Java's directlt read and wirte are too slow.
    private static StreamTokenizer in = new StreamTokenizer(new BufferedReader(new InputStreamReader(System.in)));
    private static PrintWriter out = new PrintWriter(new BufferedWriter(new OutputStreamWriter(System.out)));

    private static int nextInt() throws IOException {

        in.nextToken();
        return (int)in.nval;
    }

    private static void initializeUnionFindSet(int numberOfNodes) {

        for (int i = 0; i < numberOfNodes; i++)
            unionFindSetNodes[i] = new UnionFindSetNode(i);
    }

    private static int findRoot(int index) {

        if (unionFindSetNodes[index].parent != index) {
            int curAncestor = unionFindSetNodes[index].parent;  // save current parent, cause 
                                                                // next line it will be change to root  
            unionFindSetNodes[index].parent = findRoot(unionFindSetNodes[index].parent);    // do path compression
            unionFindSetNodes[index].relationship = (unionFindSetNodes[index].relationship
                    + unionFindSetNodes[curAncestor].relationship) % 3; // calulate the releationship with root
        }
        return unionFindSetNodes[index].parent; // now parent have been changed to root, return the root
    }

    public static void main(String[] args) throws IOException {

        int n, k, cmd, x, y, ans = 0;
        n = nextInt();
        k = nextInt();
        initializeUnionFindSet(n);
        for (int i = 0; i < k; i++) {
            cmd = nextInt();
            x = nextInt();
            y = nextInt();
            if (x > n || y > n) {
                ans++;
                continue;
            } else if (cmd == 2 && x == y) {
                ans++;
                continue;
            }
            x--;    // nodes array numbers from 0
            y--;
            cmd--;  // 1->0, 2->1, correspond with node's relationship

            int rootX, rootY;
            // find each node's root, do path compression, and caculate the relationship with root
            rootX = findRoot(x);
            rootY = findRoot(y);
            if (rootX == rootY) {
                // determine if there's contradiction
                int curRelationship = (unionFindSetNodes[x].relationship + 3 - unionFindSetNodes[y].relationship) % 3;
                if (curRelationship != cmd) {
                    ans++;
                }
            } else {
                // union two nodes
                unionFindSetNodes[rootX].parent = rootY;
                unionFindSetNodes[rootX].relationship = (3 - unionFindSetNodes[x].relationship + cmd
                        + unionFindSetNodes[y].relationship) % 3;
            }
        }

        out.println(ans);
        out.flush();
    }
}
~~~

### [POJ1125]
**[👉CLICK HERE](http://poj.org/problem?id=1125)**
#### Description
Given a graph with no more than $100$ vertixs, and edges' weight are less than $10$. Your mission is to work out **if there is a vertix can reach all other vertixs and the longest path between this vertix and others is the smallest**. If there's no vertix can reach all other vertixs, print "disjoint".

#### Analysis
Since the data range is small, consider the case of **enumerating each vertix**. For each vertix of the enumeration, you need to calculate the shortest path with the other vertixs. Therefore, **Floyd algorithm** can be used to preprocess the shortest path between any two vertixs. Then for each vertix, **calculate the maximum value of the shortest path with all other vertixs**.

#### Algorithm
+ Floyd algorithm preprocesses the shortest path between any two vertixs.
+ Enumerate each vertix and find the maximum value that is the shortest with all other vertixs. If there is a vertix that cannot be reached, enumerate the next vertix.
+ The final answer is the minimum of all enumeration results. If no vertixs can reach all vertixs, print "disjoint".

#### Code(Java)
~~~Java
import java.util.Scanner;
import java.lang.Math;

public class Main {

    private static final int INF = 0x3f3f3f3f;
    private static final int SIZE = 105;
    private static int[][] map = new int[SIZE][SIZE];
    private static int[][] dis = new int[SIZE][SIZE];
    private static int numberOfVertix;
    private static int n;

    private static Scanner sc = new Scanner(System.in);

    private static void Floyd() {

        for (int i = 0; i < SIZE; i++)
            for (int j = 0; j < SIZE; j++)
                map[i][j] = INF;
        // if there is no way between A and B, map[A][B] set as INF
        for (int i = 1; i <= n; i++) {
            int t, x, s;
            t = sc.nextInt();
            while (t-- > 0) {
                x = sc.nextInt();
                s = sc.nextInt();
                map[i][x] = s;
            }
        }

        // Floyd algorithm
        for (int t = 1; t <= n; t++)
            for (int i = 1; i <= n; i++)
                for (int j = 1; j <= n; j++)
                    map[i][j] = Math.min(map[i][j], map[i][t] + map[t][j]);
    }

    public static void main(String[] args) {

        while (sc.hasNextInt()) {
            n = sc.nextInt();
            if (n == 0)
                break;

            Floyd();
            int ans = INF, cur = 0;
            for (int i = 1; i <= n; i++) {
                // get the maximum length between i and other vertixs
                int tmp = 0;
                for (int j = 1; j <= n; j++) {
                    if (i == j)
                        continue;
                    if (map[i][j] > tmp)
                        tmp = map[i][j];
                }
                // compare with current ans, if smaller then save it
                if (ans > tmp) {
                    ans = tmp;
                    cur = i;
                }
            }

            if (ans == INF)
                System.out.println("disjoint");
            else 
                System.out.printf("%d %d\n", cur, ans);
        }
    }
}
~~~
